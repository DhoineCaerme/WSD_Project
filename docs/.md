# Session 4 – Cache and Deployment

# What tasks.index caches

The `tasks.index` key stores the complete list of all tasks returned by the
`GET /api/tasks` endpoint. It is cached in Redis for 60 seconds using
`Cache::remember('tasks.index', 60, function() {...})`.

This means the first request (cold read) queries the PostgreSQL database,
stores the result in Redis under the key `tasks.index`, and returns it.
All subsequent requests within 60 seconds (warm reads) skip the database
entirely and read directly from Redis memory, which is significantly faster.
# Why store, update, and destroy call Cache::forget('tasks.index')

Whenever a task is created (POST), updated (PUT/PATCH), or deleted (DELETE),
the cached list stored under `tasks.index` becomes outdated — it no longer
reflects the real state of the database.

`Cache::forget('tasks.index')` removes this stale cached value immediately.
The next GET request will then query PostgreSQL fresh, get the updated list,
and store it in Redis again.

This pattern is called **cache invalidation** — making sure cached data never
stays outdated after a write operation. Without this, users could see old
deleted tasks or miss newly created ones for up to 60 seconds.

> Real life analogy: Imagine a restaurant menu board (cache). When the chef
> adds a new dish (POST), the waiter erases the old board (Cache::forget) so
> the next customer sees the updated menu instead of the old one.

# Purpose of Redis in this stack

Redis acts as a high-speed in-memory cache layer sitting between the Laravel
backend and the PostgreSQL database. Its purpose is to reduce database load
and improve response times for repeated read operations.

Reading data from Redis (stored in RAM) is orders of magnitude faster than
reading from PostgreSQL (stored on disk). In this stack Redis is configured
as database 1 (`REDIS_CACHE_DB=1`) specifically for cache storage, keeping
it separate from other Redis data.

Without Redis, every single GET /api/tasks request would hit PostgreSQL
directly — even if the data hasn't changed at all since the last request.
With Redis, only the first request after an invalidation hits the database.

# Purpose of Nginx in this stack

Nginx serves two roles in this stack:

**1. Static file server** — Nginx serves the compiled Vue.js frontend files
(HTML, CSS, JavaScript) from `/usr/share/nginx/html`. When a user opens
the browser, Nginx delivers these files instantly.

**2. Reverse proxy** — Any request starting with `/api/` is automatically
forwarded (proxied) to the Laravel backend container running on port 8000.
This means both the frontend and the API are accessible through a single
port (8080) without the browser ever knowing there are two separate services
behind it.

Nginx also handles the SPA (Single Page Application) routing by falling back
to `index.html` for any unknown URL, which allows Vue Router to handle
navigation on the client side.

# Commands used to verify cache behavior

# Verify Redis is running:
```bash
docker compose exec redis redis-cli PING
# Expected: PONG
```

# Check Redis is empty before caching:
```bash
docker compose exec redis redis-cli -n 1 DBSIZE
# Expected: 0
```

# Populate the cache by hitting the task list:
```bash
curl http://127.0.0.1:8080/api/tasks
docker compose exec redis redis-cli -n 1 DBSIZE
# Expected: 1 (tasks.index key stored in Redis)
```

# Invalidate the cache by creating a new task:
```bash
curl -X POST http://127.0.0.1:8080/api/tasks \
  -H "Content-Type: application/json" \
  -d '{"title":"Created after caching","description":"invalidate test","status":"todo","priority":"medium","album_number":"78942"}'

docker compose exec redis redis-cli -n 1 DBSIZE
# Expected: 0 (cache invalidated by Cache::forget)
```

# Repopulate the cache:
```bash
curl http://127.0.0.1:8080/api/tasks
docker compose exec redis redis-cli -n 1 DBSIZE
# Expected: 1 (cache repopulated)
```

### Cold vs warm read speed comparison:
```bash
# Cold read (hits PostgreSQL)
Measure-Command { curl http://127.0.0.1:8080/api/tasks }

# Warm read (hits Redis)
Measure-Command { curl http://127.0.0.1:8080/api/tasks }
```
The warm read should complete faster than the cold read because Redis
serves the response directly from memory without querying the database.
